# Project Brief: Industrial Predictive Maintenance Platform

The document you read before writing a single line of code, so that three weeks from
now you remember what you were actually trying to do. Written in the repo voice: dry,
honest, no decoration.

## The problem

Unplanned equipment failure is expensive and stupid in the specific way that
preventable things are. Two existing strategies, both bad: run the machine until it
breaks (reactive, catastrophic), or replace parts on a fixed calendar whether they
need it or not (wasteful, ignores actual health). Predictive maintenance is the third
option: watch the machine, learn what healthy looks like, and predict failure early
enough to act but late enough that you are not replacing good parts.

We are building a platform that ingests engine sensor streams, flags anomalies before
failure, predicts how many cycles of life remain, classifies the operating regime and
degradation stage, and surfaces all of it to a maintenance engineer on a dashboard.
It is a system with several cooperating models and a real streaming pipeline, not a
single model in a notebook with a nice accuracy number.

## The dataset

NASA C-MAPSS Turbofan Engine Degradation Simulation. It is the standard benchmark in
the prognostics and health management community, which means it is well understood,
well documented, and respected by anyone who would interview you about this project.
The data is synthetic, generated by NASA's C-MAPSS simulator, but it is treated as the
real-world benchmark and hundreds of published papers use it.

Primary download (Kaggle):
https://www.kaggle.com/datasets/behrad3d/nasa-cmaps

Alternates if Kaggle is being Kaggle:
- NASA Prognostics Data Repository: https://www.nasa.gov/intelligent-systems-division (PCoE prognostic datasets)
- NASA open data mirror: https://data.nasa.gov/dataset/cmapss-jet-engine-simulated-data

### What is in it

Each engine starts healthy with some random initial wear, runs until it fails, and
records sensor values every cycle. The data is contaminated with sensor noise on
purpose, because real sensors lie a little. There are four subsets of increasing
difficulty:

| Subset | Operating conditions | Fault modes | Train engines | Test engines |
|--------|---------------------|-------------|---------------|--------------|
| FD001  | 1                   | 1 (HPC degradation) | 100   | 100          |
| FD002  | 6                   | 1 (HPC degradation) | 260   | 259          |
| FD003  | 1                   | 2 (HPC, Fan)        | 100   | 100          |
| FD004  | 6                   | 2 (HPC, Fan)        | 248   | 249          |

Start with FD001 to get the whole pipeline working end to end, because it is the
simplest. Then move to FD004, which is the hardest (six conditions, two fault modes)
and the one that makes the project interesting. Building only on FD001 and stopping is
the lazy version that interviewers have seen a hundred times. The whole point of this
project is that you went further.

### File structure

Each subset ships three space-separated text files:

- `train_FD00X.txt`: full run-to-failure trajectories. Every engine runs until it
  dies, so the last cycle for each engine is the failure point.
- `test_FD00X.txt`: trajectories that stop some time before failure. Your job is to
  predict how much life was left.
- `RUL_FD00X.txt`: the ground-truth remaining useful life for each test engine, so you
  can score your predictions.

### Data dictionary

26 columns, no header, space separated. In order:

| Column | Name              | Meaning |
|--------|-------------------|---------|
| 1      | unit_number       | Engine id. Groups rows into one engine's life. |
| 2      | time_cycles       | Operational cycle count. The clock. |
| 3      | op_setting_1      | Operational setting. In multi-condition subsets these define the regime. |
| 4      | op_setting_2      | Operational setting. |
| 5      | op_setting_3      | Operational setting. |
| 6 to 26| sensor_1 .. sensor_21 | 21 sensor channels: temperatures, pressures, shaft speeds, flow ratios. Unlabeled by design. |

The sensors are deliberately anonymous. You do not get told which is a temperature and
which is a pressure. That is part of the challenge: you find the informative ones
yourself. Several sensors are flat constants in FD001 and carry no signal; dropping
them is feature engineering, not cheating.

### The honest note about the third model

The original three-model pitch was anomaly detection, RUL prediction, and fault
classification. Two of those map cleanly onto this data. The third needs a reframing,
and pretending otherwise would be the kind of thing a sharp interviewer catches.

- Anomaly detection: clean fit. Train an autoencoder on early-life (healthy) sensor
  windows, flag reconstruction error as the engine degrades. Unsupervised, honest, no
  label needed.
- RUL prediction: clean fit. This is literally what the dataset is built for. The
  ground truth is provided.
- The "fault classifier": C-MAPSS does not give you a per-row fault-type label. FD003
  and FD004 contain two fault modes, but the data does not tell you which engine has
  which. So do not claim to classify fault type from labels that do not exist. Instead,
  pick one of these defensible framings and say which you chose and why:
  - Operating-condition classification: in FD002 and FD004 there are six operating
    regimes encoded in the operational settings. Classifying the regime from sensor
    data is a genuine, honest multiclass problem that the serving system actually
    needs, because models behave differently per regime.
  - Degradation-stage classification: bucket each cycle into healthy, degrading, or
    critical using RUL thresholds, and classify the stage. This feeds the dashboard
    directly and is the kind of derived label that real systems use.

Either one is a stronger story than a fake fault classifier, because you can explain
the labeling honestly. Honesty about data is itself a senior-engineer signal.

## What we are building

A system with four parts that talk to each other:

1. Ingestion: a streaming path that replays sensor data through Pub/Sub as if it were
   live, processed by Dataflow, so the system handles a real stream and not just a
   static file. This is the component most portfolios lack and the one that makes
   yours stand out.
2. Models: the anomaly autoencoder, the RUL regressor, and the regime or
   degradation-stage classifier, sharing one feature implementation.
3. Serving: real-time predictions on a Vertex AI Endpoint, plus batch prediction for
   bulk scoring.
4. Monitoring and dashboard: drift detection on the inputs and predictions, alerting,
   and a view an engineer actually looks at.

## Architecture and GCP services

- Stream simulation: Pub/Sub publishes sensor records, Dataflow consumes and processes
  them in real time.
- Data versioning: DVC with a GCS remote. Every model traces to its exact training
  data.
- Training orchestration: Vertex AI Pipelines (Kubeflow underneath). Each step is a
  container: validate data, engineer features, train, evaluate, register.
- Experiment tracking: Vertex AI Experiments, logging the full contract from the
  experiment-logging skill.
- Model registry: Vertex AI Model Registry with a promotion gate. A model reaches
  production by beating the incumbent on the metric, not by vibes.
- Serving: Vertex AI Endpoints (real time) and Batch Prediction (bulk).
- Monitoring: Vertex AI Model Monitoring for drift, Cloud Logging and Monitoring for
  latency and errors.
- Storage and warehouse: GCS for artifacts and data, BigQuery for predictions and
  metrics.
- CI/CD: Cloud Build triggered by GitHub, gating deploy on data and model quality.
- Frontend: Cloud Run serving the dashboard.

The full diagram belongs in docs/architecture.md. This brief is the why, that doc is
the how.

## The metric that actually matters

For RUL, do not optimize plain RMSE and call it a day. In maintenance, predicting
failure late is much worse than predicting it early: an early prediction wastes a
little life, a late one means the engine failed in service. Use an asymmetric scoring
function that penalizes late predictions harder, which is the standard C-MAPSS scoring
function from the original PHM08 challenge. State this in every model card and every
evaluation. It is the difference between someone who ran a regression and someone who
understood the problem.

## Build plan

Each phase is a milestone. Each task is a GitHub issue using the templates. Do not
start a phase by writing code, start it by writing the issues.

### Phase 0: Foundations
- Repo scaffolding, the .claude harness (done), CI skeleton.
- DVC remote wired to a GCS bucket, FD001 added and versioned.
- A loader that parses the space-separated files into a clean, typed dataframe with
  the column names from the data dictionary.
- Definition of done: someone can clone the repo, pull the data, and load FD001 in one
  documented command.

### Phase 1: Single model, end to end, on FD001
- Feature engineering shared between train and serve. Drop the dead sensors, add
  rolling windows, normalize per regime.
- The RUL regressor first, because the label is handed to you. Baseline, then a real
  model. Log everything.
- A Vertex AI Pipeline that trains, evaluates against the asymmetric metric, and
  registers if it passes the gate.
- Definition of done: a registered RUL model with a model card and a reproducible
  pipeline run.

### Phase 2: The other two models
- Anomaly autoencoder on healthy windows, with a sensible reconstruction-error
  threshold and a justification for it.
- Regime or degradation-stage classifier, with the labeling choice documented per the
  honest note above.
- Definition of done: three registered models, three model cards, one shared feature
  pipeline.

### Phase 3: Serving
- Deploy the models to Vertex AI Endpoints. Wire batch prediction for bulk scoring.
- A prediction service that takes a sensor window and returns anomaly score, RUL, and
  stage together.
- Definition of done: a live endpoint, a documented request and response shape, a
  rollback command.

### Phase 4: Streaming
- Pub/Sub publisher that replays test trajectories as a live stream.
- Dataflow job that consumes, builds features, calls the endpoint, and writes results
  to BigQuery.
- Definition of done: data flows from simulated sensor to stored prediction with no
  manual step.

### Phase 5: Monitoring and dashboard
- Vertex AI Model Monitoring on input and prediction distributions. Alert thresholds
  that mean something. Runbooks for when they fire.
- A Cloud Run dashboard showing live engine health, RUL, and anomalies.
- Definition of done: the system tells a human before a user would notice, and the
  human has instructions.

### Phase 6: Hardening and scale
- Move from FD001 to FD004. Handle the six operating conditions and two fault modes.
- Document what broke when the data got harder, because something will, and the
  write-up is the interesting part.
- Definition of done: the pipeline runs on FD004 and the README tells the honest story
  of what the added complexity cost.

## What good looks like

A stranger lands on this repo and within a minute understands the problem, sees a clean
architecture diagram, finds a README that actually explains how to run it, notices
that every model has a card, sees that experiments are tracked and data is versioned,
and concludes that the person who built this knew what they were doing. That is the
target. Not the highest accuracy. The most credible system.

## Getting started

1. Clone the repo and read CLAUDE.md and .claude/README.md.
2. Create the GCS bucket and wire the DVC remote (see the dvc-data-versioning skill).
3. Download the dataset from the Kaggle link above. With the Kaggle CLI:
   `kaggle datasets download -d behrad3d/nasa-cmaps`
4. Add it to DVC and commit the pointer, not the data.
5. Open the Phase 0 issues using the templates and start there.
6. Use /train, /evaluate, and /ship as you go. Let the hooks keep you honest.
